# Qwen3-32B

Standard **Qwen3 dense** model, 32B parameters, served in **BF16** (~62 GB across
four cards). Validated on **Ascend 910B4 (32 GB/card)** with the stock vLLM-Ascend
**v0.18.0** engine. The validated 8-card topology is **TP=4 Г— 2 replicas** behind
load-aware routing вЂ” two TP=4 instances, a half-scale reproduction of the *InferNex
v26.06* agg-base baseline (16 cards).

:::note
Qwen3-32B is a standard dense model, so the stock vLLM-Ascend `v0.18.0` image loads
it directly вЂ” no nightly engine and no speculative (MTP) head. Prefix caching is on
(`--enable-prefix-caching`), which lowers TTFT on workloads that reuse a shared
system prompt. The Ascend toolkit env must be sourced before `vllm serve`, or the TP
workers crash on `libatb`; that sourcing is baked into the `ServingRuntime` command.
:::

output pinned to 128 tokens. TTFT / E2E in seconds unless noted, ITL in ms, TPS in

:::note
**How to read these.** Both scenarios completed 240/240 with zero errors at a steady
2 in-flight requests per instance. ITL (per-token, no MTP on a dense model) is ~71 ms
on the 8k workload and ~96 ms on the 17.5k workload вЂ” longer context means a heavier
decode step. TTFT on scenario в‘  stays near 1.3 s because prefix caching reuses the
shared 8k system prompt; on scenario в‘Ў, growing per-turn context drives TTFT up and
its p90 spreads as some turns are re-prefilled after cache eviction. TPS is the
total-token (input + output) caliber; the decode-only figure is the system's true
output rate and is naturally much lower under these long-input workloads. These are
half-scale numbers (8 cards); aggregate throughput scales with the instance count.
:::
