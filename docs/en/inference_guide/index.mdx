# Inference Guide

Ready-to-deploy recipes for **validated open-weight LLMs** on Alauda AI. Each model
in this guide has been deployed end-to-end on a real cluster and benchmarked, so you
get a known-good `ServingRuntime` + `InferenceService` pair, the runtime image that
serves it, and the throughput you can expect.

The models here were validated on **Huawei Ascend 910B4 NPU** with the community
`vLLM-Ascend` engine. Both were run in the **same aggregation topology and the same
benchmark caliber**, so their results are directly comparable. For the runtime model
(KServe `ServingRuntime` / `InferenceService`, ModelCar storage, scheduling) see
[Model Deployment & Inference](../model_inference/index.mdx); for adding your own
engine see [Extend Inference Runtimes](../model_inference/inference_service/how_to/custom_inference_runtime.mdx).

load-aware routing. This is a half-scale reproduction of the *InferNex v26.06*
16-card baseline, so per-instance latency is directly comparable and aggregate
throughput scales with the instance count.

## Benchmark methodology

All numbers in this guide come from real `aiperf` runs on the validated cluster, not
estimates. Both models were measured with an identical setup:

- **Topology** — agg-base aggregation, **2 instances × TP=4 = 8 Ascend 910B4 cards**
  (one node), each instance a vLLM-Ascend engine behind load-aware routing. This is a
  half-scale reproduction of the *InferNex v26.06* 16-card baseline (4 × TP=4); latency
  is directly comparable, aggregate throughput scales with the instance count.
- **Closed-loop, concurrency 4** (`aiperf --concurrency 4`, 2 in-flight requests per
  instance). This is a deliberate **exception** to a pure open-loop sweep, chosen to
  reproduce the reference report's operating point apples-to-apples.
- **Two workloads** (ISL/OSL tokens): ① **8k system-prompt reuse** `8000/128` and
  ② **17.5k multi-turn** `~17.9k/128`. 240 requests each, output pinned to 128 tokens.
- **TPS = total-token throughput (input + output)** = (Σ input + Σ output) / wall-time,
  matching the reference caliber. The **decode-only** output throughput is reported
  separately and is much smaller under these long-input workloads — don't confuse the two.

:::warning Not a strict A/B between the two models
Each model runs its own best validated config: Qwen3.6-27B **requires**
`--no-enable-prefix-caching` (the GDN hybrid cache), while Qwen3-32B runs with
`--enable-prefix-caching`. The Qwen3.6-27B advantage therefore reflects the
*combination* of W8A8 + MTP + fewer parameters, not quantization alone — and on the
8k-reuse workload, prefix caching gives Qwen3-32B a lower mean TTFT.
:::

## Benchmark summary

Closed-loop, concurrency 4, agg-base 2 × TP=4 (8 × 910B4), `aiperf 0.7.0`. TTFT / E2E
mean, ITL per-chunk (Qwen3.6-27B MTP emits ~3 tokens/chunk), TPS = total tokens/s.

**Scenario ① — 8k system-prompt reuse, output 128**

| Model | dtype | ITL avg (ms) | ITL p90 | TTFT avg (ms) | E2E avg (s) | TPS (in+out) | Decode tok/s | Success |
|---|---|--:|--:|--:|--:|--:|--:|--:|
| Qwen3.6-27B | W8A8 | 32.0 | 45.0 | 1509 | 5.6 | 5807 | 91.3 | 240/240 |
| Qwen3-32B | BF16 | 71.4 | 84.6 | 1328 | 10.4 | 3120 | 49.1 | 240/240 |

**Scenario ② — 17.5k multi-turn, output 128**

| Model | dtype | ITL avg (ms) | ITL p90 | TTFT avg (ms) | E2E avg (s) | TPS (in+out) | Decode tok/s | Success |
|---|---|--:|--:|--:|--:|--:|--:|--:|
| Qwen3.6-27B | W8A8 | 45.3 | 78.4 | 3999 | 9.8 | 7408 | 52.5 | 240/240 |
| Qwen3-32B | BF16 | 95.9 | 135.2 | 4612 | 16.8 | 4270 | 30.3 | 240/240 |

At a matched operating point, Qwen3.6-27B (W8A8) serves at roughly **half the
inter-token latency** and **~1.7–1.9× the total throughput** of Qwen3-32B (BF16) —
the payoff of fewer parameters, INT8 weights, and the native MTP speculative head.
Per-model detail and percentiles are on each model's page.

## Deploy a validated model

Each model page links a self-contained YAML under
[`assets/`](https://github.com/alauda/aml-docs/tree/master/docs/en/inference_guide/assets)
that bundles a namespace-scoped `ServingRuntime` **and** the matching `InferenceService`
(2 replicas × TP=4 = 8 cards).

```bash
base=https://raw.githubusercontent.com/alauda/aml-docs/master/docs/en/inference_guide/assets

# 1. Edit the file first: set metadata.namespace, the image tag (match your CANN /
#    host driver), and storageUri (a PVC of the weights, or an oci:// ModelCar).
# 2. Apply the ServingRuntime + InferenceService.
kubectl apply -f $base/qwen3-32b/qwen3-32b-vllm-ascend-tp4x2.yaml

# 3. Watch it come up (first start loads weights + compiles graphs — can take minutes).
kubectl -n <your-namespace> get inferenceservice -w

# 4. Call the OpenAI-compatible endpoint.
curl -s http://<isvc-url>/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"qwen3-32b","messages":[{"role":"user","content":"hello"}]}'
```

## Caveats

- These are **namespace-scoped `ServingRuntime`** examples (not `ClusterServingRuntime`),
  one per model — apply them in the same namespace as the `InferenceService` that
  references them.
- Resource keys are for **Ascend 910B4** (`huawei.com/Ascend910`). Adjust the resource
  key, image, and version fields for your actual NPU model.
- For Ascend single-node **multi-card** (TP>1) the runtime image must support HCCL init
  under the configured Modelcar UID — see the Modelcar permission modes in
  [Extend Inference Runtimes](../model_inference/inference_service/how_to/custom_inference_runtime.mdx#modelcar-permission-modes-for-ascend-910).
- The benchmark numbers were measured **closed-loop** (a deliberate exception to a pure
  open-loop sweep) to reproduce a reference report apples-to-apples. Treat them as the
  per-instance operating point under steady 2-in-flight load, not a saturation ceiling.
