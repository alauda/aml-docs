解读 · 数据告诉我们什么

读数注意（这些是事实，不是结论）：
  • 表里 RPS mean/mdn 全 0.000 不代表 0 吞吐，而是 1s bucket 在 ~0.03 RPS 下的统计 artifact（每秒大多时候是 0 个完成事件，偶尔 1 个，中位/均值都 round 到 0）。真实有效完成 RPS = comp/dur ≈ 0.022-0.031（见 KPI 卡片）。✅实测 + 🧮 可靠推算
  • Inc（incomplete）随 rate 线性涨（rate=1: 61 → rate=9: 507），Output Err 全程 0（server 没有 streaming 失败）。Input Err 显示 ~28 req/档 是 client 端 60s timeout（见 log line 7 `'timeout': 60.0`）超过 E2E p95 52-77s 的尾部所致——不是 server 拒绝，server 端 max-num-seqs=2 不会显式拒绝，只会让请求在 chunked-prefill 队列里继续等。✅实测
  • ITL p50 ≈ 200ms 跨 rate 几乎不动（181-218ms），证明 decode 速率与 client 端压力无关，只取决于 server 端单 step kernel 时间。这是"per-request 用户体验"指标，正常 ≈ 5 tok/s 输出。✅实测
  • TTFT p95 在 rate=1 飚到 22s 是 cold-start 余波（391GB 权重首次推理还在 warmup 后段做 kernel select/cache）。rate 2-9 的 TTFT p95 都在 1.1-1.6s 区间，是真实稳态。✅实测
  • TPOT ≈ ITL：vllm-ascend 在 enforce-eager 下没有 cudagraph 加速，TPOT 跟 ITL 几乎一致（无 prefetch / kv-cache overlap 优化）。开 cudagraph 理论上 TPOT 可降 20-30%，但本次走 eager 是为了绕开 aclnnDynamicQuantV2 算子崩溃，perf 代价已纳入此次测量。✅实测 + 🧮 推算
  • 单流权重读取 ≈ 105 GB/s（5 tok/s × 21 GB/token 激活权重）。📋 厂商规格 910B3 单卡 HBM 800 GB/s × 8 = 6.4 TB/s 总带宽——单流只用了约 1.6% 的总带宽（受 enforce-eager + 单 stream 串行限制），带宽远未跑满。这是为什么提 max-num-seqs 仍有空间。🧮 可靠推算
对"让模型服务更多人"目标的判断：
  • 当前配置稳态 2 并发不是硬件瓶颈，是 --max-num-seqs=2 这个 v0.18 doc A2 w4a8 推荐值——doc 写"A2 显存吃紧"，但贵安 64GB 单卡 + W4A8 量化后 KV cache 还有 6.36 GB 余量，理论上 max-num-seqs=8-16 是可达的。下一步该做的实验是：抬 max-num-seqs 到 4 / 8 / 16，看 (a) 还能不能 cold-load，(b) ITL 是否随并发劣化，(c) 聚合 TPS 能不能线性扩。🧮 可靠推算 + ⚠️ 待验证
  • 当前数据的"生产容量"诚实表述：在 vllm-ascend v0.19.1rc1 + enforce-eager workaround + max-num-seqs=2 下，该 ISVC 可承载 2 路同时聊天（每路 ~5 tok/s 输出 / E2E ~52s @ 256 tok）。要超出 2 路需要先验证更高 max-num-seqs 在贵安环境下不触发新的算子崩。
  • 关于 enforce-eager 的代价：上游 doc 在 PD-disaggregation prefill 节点（大 prefill 场景）就用 enforce-eager，所以这不是"实验性 hack"，是 doc 认可的 workaround。代价仅在 decode（typical 20-30% ITL 涨），TTFT 影响小。🧮 推算 + 📋 上游 doc 背书

背景上下文

背景（这次为什么是 enforce-eager workaround）：
  • 05 Chatbot 首跑（prompt=512） 触发 aclnnDynamicQuantV2 AI Core 异常（err 507011 / EZ9999 fftsplus / aic_error_mask=0x6500020bd00028c / pc=0x12c09e3a0ba8），predictor pod crash。
  • 假设 A（已证伪）：v0.19.1rc1 的 norm_quant_fusion_pass.py output_mask=[True,True] bug（upstream 已修为 [True,False]）。关 fuse_norm_quant 重跑——同一个 pc / 同一个 mask 崩，bug 真实存在但非本次崩溃因果起点。
  • 假设 B（本次走通）：直接退到 --enforce-eager，绕开编译路径生成的 kernel binary，让算子走纯 eager 直派 aclnn op。代价：decode 丢 cudagraph 加速（典型 ITL +20-30%）。上游 doc 在 PD-prefill 节点同样用 enforce-eager，有先例。
  • 结果：9-rate 300s × 9 sweep 全程 0 err，无 crash，pod 全程 1/1 Running。enforce-eager 是 doc 认可的退路，不是 hack。
  • 不可达 / 已排除路径：(a) 升级镜像到 main → v0.19.1rc1 已是 latest tag，main 无针对此 op 的修复；(b) 换 W8A8 → ~755GB > 512GB 硬件 HBM，装不下；(c) 厂商 escalation → 周/月级时间窗，不适用当下交付。
